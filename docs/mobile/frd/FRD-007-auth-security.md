# FRD-007 — Authentication & Security

**Status:** Draft
**Priority:** High
**Owner:** Scotty
**Implements:** PRD constraints (security)
**Depends On:** FRD-001

---

## Purpose

Define how the iOS client authenticates with World Tree's server and how the connection is secured. The existing CanvasServer uses a shared token (`x-canvas-token`). This FRD extends that model for WebSocket connections and addresses transport security for both local and remote scenarios.

## User Stories

- As a mobile user, I want my connection to be authenticated so unauthorized devices can't access my conversations.
- As a mobile user, I want to pair my iPhone with my Mac once and not re-enter credentials.
- As a user, I want my conversation data encrypted in transit when connecting remotely.

## Functional Requirements

### Token Authentication

**FR-007-001:** The iOS client SHALL authenticate WebSocket connections by including the server token as a query parameter: `ws://host:port/ws?token=xxx`.

**FR-007-002:** The server token SHALL be the same `cortana.serverToken` used by existing HTTP endpoints.

**FR-007-003:** If the token is missing or invalid during WebSocket upgrade, the server SHALL respond with HTTP 401 and not complete the upgrade.

**FR-007-004:** The token SHALL be stored in the iOS Keychain, associated with the server's saved ID.

**FR-007-005:** The settings screen SHALL provide a "Token" field when adding or editing a server, with a paste button and secure text field (dots, not plain text).

### Initial Pairing

**FR-007-006:** For first-time setup, the macOS app SHALL display the server token in Settings → Server, with a "Copy" button.

**FR-007-007:** The iOS app SHALL provide a manual token entry field during server setup.

**FR-007-008:** (Future) QR code pairing: Mac displays QR containing `{host, port, token}`, iPhone scans. Out of scope for v1 but the data model supports it.

### Transport Security

**FR-007-009:** On local network (Bonjour), connections use `ws://` (unencrypted). This is acceptable because:
- Traffic stays on the local network
- The auth token prevents unauthorized access
- mDNS does not expose the token

**FR-007-010:** On Tailscale, connections use `ws://` over WireGuard tunnel. Tailscale provides end-to-end encryption at the network layer. No application-level TLS needed.

**FR-007-011:** The server SHALL NOT support `wss://` (WebSocket Secure) in v1. TLS termination is handled by the network layer (Tailscale) or is unnecessary (local).

**FR-007-012:** If a future version requires direct internet exposure (no Tailscale), TLS SHALL be added at that time. The architecture must not preclude this.

### Token Rotation

**FR-007-013:** The macOS Settings → Server SHALL include a "Regenerate Token" button that creates a new random token.

**FR-007-014:** Token regeneration SHALL immediately disconnect all active WebSocket clients (they'll fail to reconnect until re-paired).

**FR-007-015:** The server SHALL log token validation failures (IP, timestamp) for security awareness.

### Connection Security

**FR-007-016:** The server SHALL rate-limit failed authentication attempts: max 5 failures per IP per minute. Further attempts receive HTTP 429.

**FR-007-017:** The server SHALL not include the token in any log output.

**FR-007-018:** The iOS client SHALL not include the token in any log output or crash reports.

## Data Requirements

**macOS (existing):**
- `UserDefaults["cortana.serverToken"]` — shared server token (existing)

**iOS (new):**
- Keychain: `server.{serverId}.token` — per-server auth token
- No additional storage needed

## Business Rules

- BR-001: One token per server. All clients (Telegram, iOS, future) use the same token.
- BR-002: Token is a random 32-character hex string (generated by macOS app).
- BR-003: Token is never transmitted over Bonjour discovery (only over established connections).
- BR-004: Security model assumes trusted local network or encrypted tunnel (Tailscale). Not designed for open internet.
- BR-005: Rate limiting is per-IP, not per-token (prevents brute force from single source).

## Error States

| Error | Response | Recovery |
|-------|----------|----------|
| Invalid token on connect | 401 + connection refused | Re-enter token in iOS settings |
| Token rotated on Mac | All iOS clients disconnect | Re-pair with new token |
| Rate limited (429) | "Too many attempts, try again in 60s" | Wait and retry |
| Keychain access denied | "Cannot access stored credentials" | Re-enter token; check iOS permissions |

## Acceptance Criteria

1. WebSocket connection succeeds only with valid token
2. Invalid token returns 401, connection not established
3. Token stored in iOS Keychain (not UserDefaults)
4. Token not visible in any logs (server or client)
5. Rate limiting blocks brute force attempts (>5 failures/min/IP)
6. Token regeneration on Mac disconnects all mobile clients
7. Tailscale connection works without additional TLS configuration

## Out of Scope

- TLS/WSS (handled by network layer)
- Certificate pinning
- OAuth / external auth providers
- Biometric unlock for the iOS app
- QR code pairing (v2)
- Per-client tokens (all clients share one token for v1)
- End-to-end encryption above transport layer

## Technical Notes

### Token in WebSocket Upgrade

`URLSessionWebSocketTask` supports custom headers via `URLRequest`, but some WebSocket implementations strip custom headers. The query parameter approach (`?token=xxx`) is more reliable:

```swift
var components = URLComponents()
components.scheme = "ws"
components.host = server.host
components.port = Int(server.port)
components.path = "/ws"
components.queryItems = [URLQueryItem(name: "token", value: token)]
let task = URLSession.shared.webSocketTask(with: components.url!)
```

On the server side, parse the query string from the HTTP upgrade request path.

### Rate Limiting Implementation

Simple in-memory rate limiter on CanvasServer:

```swift
private var authFailures: [String: (count: Int, windowStart: Date)] = [:]

func checkRateLimit(ip: String) -> Bool {
    let now = Date()
    if let record = authFailures[ip] {
        if now.timeIntervalSince(record.windowStart) > 60 {
            authFailures[ip] = (1, now)
            return true
        }
        if record.count >= 5 { return false }
        authFailures[ip]?.count += 1
    } else {
        authFailures[ip] = (1, now)
    }
    return true
}
```
